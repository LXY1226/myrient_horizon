# Generated vs Handwritten Code Boundaries

## Generated Code (READ-ONLY - Never Edit)

Files matching these patterns are auto-generated and MUST NOT be manually modified:

### FlatBuffers Generated
- `pkg/myrienttree/MyrientTree/*.go` - All files in this directory
- **Source**: FlatBuffers `.fbs` schema files
- **Generator**: `flatc` compiler
- **Evidence**: First line contains `// Code generated by the FlatBuffers compiler. DO NOT EDIT.`

**Specific Files:**
- `pkg/myrienttree/MyrientTree/Node.go` - Generated FlatBuffers code for tree nodes

## Handwritten Shared Packages

These files are handwritten and SHOULD follow worker-derived standards where applicable:

### Protocol Package
- `pkg/protocol/messages.go` - **HANDWRITTEN**
  - Contains custom protocol message types
  - Manual JSON serialization/deserialization
  - Message type constants and structures
  - Functions: `UnmarshalConnMessage`, `UnmarshalMessage`, `MarshalConnMessage`

### Tree Package
- `pkg/myrienttree/tree.go` - **HANDWRITTEN**
  - Generic tree implementation (`Tree[DirExt, FileExt]`)
  - Custom DFS traversal logic
  - In-memory tree loaded from flatbuffer
  - Helper functions for tree navigation

## Style Application Rules

### Generated Code
- **DO NOT** modify formatting, naming, or structure
- **DO NOT** run linters or formatters on these files
- Exclude from style enforcement tools

### Handwritten Shared Packages
- Follow `backend-style.md` worker patterns where applicable
- Apply Effective Go naming conventions
- Subject to review and standardization

## Verification

To verify if a file is generated:
1. Check for `// Code generated by...` header
2. Check file location against patterns above
3. Generated files have mechanical/regular structure
